152/** Coerce the two expressions to have the same type, using C-style
153 * casting rules. For the purposes of casting, a boolean type is
154 * UInt(1). We use the following procedure:
155 *
156 * If the types already match, do nothing.
157 *
158 * Then, if one type is a vector and the other is a scalar, the scalar
159 * is broadcast to match the vector width, and we continue.
160 *
161 * Then, if one type is floating-point and the other is not, the
162 * non-float is cast to the floating-point type, and we're done.
163 *
164 * Then, if both types are unsigned ints, the one with fewer bits is
165 * cast to match the one with more bits and we're done.
166 *

168 * cast to match the one with more bits and we're done.
169 *

171 * int, both are cast to a signed int with the greater of the two
172 * bit-widths. For example, matching an Int(8) with a UInt(16) results

224/** Implementations of division and mod that are specific to Halide.
225 * Use these implementations; do not use native C division or mod to
226 * simplify Halide expressions. Halide division and modulo satisify
227 * the Euclidean definition of division for integers a and b:
228 *
229 /code
230 when b != 0, (a/b)*b + a%b = a
231 0 <= a%b < |b|
232 /endcode
233 *
234 * Additionally, mod by zero returns zero, and div by zero returns
235 * zero. This makes mod and div total functions.
236 */
237// @{
238template<typename T>
239inline T mod_imp(T a, T b) {
240 Type t = type_of<T>();
241 if (!t.is_float() && b == 0) {
242 return 0;
243 } else if (t.is_int()) {
244 int64_t ia = a;
245 int64_t ib = b;
246 int64_t a_neg = ia >> 63;
247 int64_t b_neg = ib >> 63;
248 int64_t b_zero = (ib == 0) ? -1 : 0;
249 ia -= a_neg;
250 int64_t r = ia % (ib | b_zero);
251 r += (a_neg & ((ib ^ b_neg) + ~b_neg));
252 r &= ~b_zero;
253 return r;
254 } else {
255 return a % b;
256 }
257}
258
259template<typename T>
260inline T div_imp(T a, T b) {
261 Type t = type_of<T>();
262 if (!t.is_float() && b == 0) {
263 return (T)0;
264 } else if (t.is_int()) {
265 // Do it as 64-bit
266 int64_t ia = a;
267 int64_t ib = b;
268 int64_t a_neg = ia >> 63;
269 int64_t b_neg = ib >> 63;
270 int64_t b_zero = (ib == 0) ? -1 : 0;
271 ib -= b_zero;
272 ia -= a_neg;
273 int64_t q = ia / ib;
274 q += a_neg & (~b_neg - b_neg);
275 q &= ~b_zero;
276 return (T)q;
277 } else {
278 return a / b;
279 }
280}

1150/** Linear interpolate between the two values according to a weight.
1151 * \param zero_val The result when weight is 0
1152 * \param one_val The result when weight is 1
1153 * \param weight The interpolation amount
1154 *
1155 * Both zero_val and one_val must have the same type. All types are
1156 * supported, including bool.
1157 *
1158 * The weight is treated as its own type and must be float or an
1159 * unsigned integer type. It is scaled to the bit-size of the type of
1160 * x and y if they are integer, or converted to float if they are
1161 * float. Integer weights are converted to float via division by the
1162 * full-range value of the weight's type. Floating-point weights used
1163 * to interpolate between integer values must be between 0.0f and
1164 * 1.0f, and an error may be signaled if it is not provably so. (clamp
1165 * operators can be added to provide proof. Currently an error is only
1166 * signalled for constant weights.)
1167 *
1168 * For integer linear interpolation, out of range values cannot be
1169 * represented. In particular, weights that are conceptually less than
1170 * 0 or greater than 1.0 are not representable. As such the result is
1171 * always between x and y (inclusive of course). For lerp with
1172 * floating-point values and floating-point weight, the full range of
1173 * a float is valid, however underflow and overflow can still occur.
1174 *
1175 * Ordering is not required between zero_val and one_val:
1176 * lerp(42, 69, .5f) == lerp(69, 42, .5f) == 56
1177 *
1178 * Results for integer types are for exactly rounded arithmetic. As
1179 * such, there are cases where 16-bit and float differ because 32-bit
1180 * floating-point (float) does not have enough precision to produce
1181 * the exact result. (Likely true for 32-bit integer
1182 * vs. double-precision floating-point as well.)
1183 *
1184 * At present, double precision and 64-bit integers are not supported.
1185 *
1186 * Generally, lerp will vectorize as if it were an operation on a type
1187 * twice the bit size of the inferred type for x and y.
1188 *
1189 * Some examples:
1190 * \code
1191 *
1192 * // Since Halide does not have direct type delcarations, casts
1193 * // below are used to indicate the types of the parameters.
1194 * // Such casts not required or expected in actual code where types
1195 * // are inferred.
1196 *
1197 * lerp(cast<float>(x), cast<float>(y), cast<float>(w)) ->
1198 * x * (1.0f - w) + y * w
1199 *
1200 * lerp(cast<uint8_t>(x), cast<uint8_t>(y), cast<uint8_t>(w)) ->

1503/** Scatter and gather are used for update definition which must store
1504 * multiple values to distinct locations at the same time. The
1505 * multiple expressions on the right-hand-side are bundled together
1506 * into a "gather", which must match a "scatter" the the same number
1507 * of arguments on the left-hand-size. For example, to store the
1508 * values 1 and 2 to the locations (x, y, 3) and (x, y, 4),
1509 * respectively:
1510 *
1511\code
1512f(x, y, scatter(3, 4)) = gather(1, 2);
1513\endcode
1514 *
1515 * The result of gather or scatter can be treated as an
1516 * expression. Any containing operations on it can be assumed to
1517 * distribute over the elements. If two gather expressions are
1518 * combined with an arithmetic operator (e.g. added), they combine
1519 * element-wise. The following example stores the values 2 * x, 2 * y,
1520 * and 2 * c to the locations (x + 1, y, c), (x, y + 3, c), and (x, y,
1521 * c + 2) respectively:
1522 *
1523\code
1524f(x + scatter(1, 0, 0), y + scatter(0, 3, 0), c + scatter(0, 0, 2)) = 2 * gather(x, y, c);
1525\endcode
1526*
1527* Repeated values in the scatter cause multiple stores to the same
1528* location. The stores happen in order from left to right, so the
1529* rightmost value wins. The following code is equivalent to f(x) = 5
1530*
1531\code
1532f(scatter(x, x)) = gather(3, 5);
1533\endcode
1534*
1535* Gathers are most useful for algorithms which require in-place
1536* swapping or permutation of multiple elements, or other kinds of
1537* in-place mutations that require loading multiple inputs, doing some
1538* operations to them jointly, then storing them again. The following
1539* update definition swaps the values of f at locations 3 and 5 if an
1540* input parameter p is true:
1541*
1542\code
1543f(scatter(3, 5)) = f(select(p, gather(5, 3), gather(3, 5)));
1544\endcode
1545*
1546* For more examples of the use of scatter and gather, see
1547* test/correctness/multiple_scatter.cpp
1548*
1549* It is not currently possible to use scatter and gather to write an
1550* update definition in which the *number* of values loaded or stored
1551* varies, as the size of the scatter/gather packet must be fixed a
1552* compile-time. A workaround is to make the unwanted extra operations
1553* a redundant copy of the last operation, which will be
1554* dead-code-eliminated by the compiler. For example, the following
1555* update definition swaps the values at locations 3 and 5 when the
1556* parameter p is true, and rotates the values at locations 1, 2, and 3
1557* when it is false. The load from 3 and store to 5 will be redundantly
1558* repeated:
1559*
1560\code
1561f(select(p, scatter(3, 5, 5), scatter(1, 2, 3))) = f(select(p, gather(5, 3, 3), gather(2, 3, 1)));
1562\endcode
1563*
1564* Note that in the p == true case, we redudantly load from 3 and write
1565* to 5 twice.
