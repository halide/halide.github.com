64 /** A multi-dimensional domain over which to iterate. Used when
65  * defining functions with update definitions.
66  *
67  * An reduction is a function with a two-part definition. It has an
68  * initial value, which looks much like a pure function, and an update
69  * definition, which may refer to some RDom. Evaluating such a
70  * function first initializes it over the required domain (which is
71  * inferred based on usage), and then runs update rule for all points
72  * in the RDom. For example:
73  *
74  \code
75  Func f;
76  Var x;
77  RDom r(0, 10);
78  f(x) = x; // the initial value
79  f(r) = f(r) * 2;
80  Image<int> result = f.realize(10);
81  \endcode
82  *
83  * This function creates a single-dimensional buffer of size 10, in
84  * which element x contains the value x*2. Internally, first the
85  * initialization rule fills in x at every site, and then the update
86  * definition doubles every site.
87  *
88  * One use of reductions is to build a function recursively (pure
89  * functions in halide cannot be recursive). For example, this
90  * function fills in an array with the first 20 fibonacci numbers:
91  *
92  \code
93  Func f;
94  Var x;
95  RDom r(2, 18);
96  f(x) = 1;
97  f(r) = f(r-1) + f(r-2);
98  \endcode
99  *
100  * Another use of reductions is to perform scattering operations, as
101  * unlike a pure function declaration, the left-hand-side of an update
102  * definition may contain general expressions:
103  *
104  \code
105  ImageParam input(UInt(8), 2);
106  Func histogram;
107  Var x;
108  RDom r(input); // Iterate over all pixels in the input
109  histogram(x) = 0;
110  histogram(input(r.x, r.y)) = histogram(input(r.x, r.y)) + 1;
111  \endcode
112  *
113  * An update definition may also be multi-dimensional. This example
114  * computes a summed-area table by first summing horizontally and then
115  * vertically:
116  *
117  \code
118  ImageParam input(Float(32), 2);
119  Func sum_x, sum_y;
120  Var x, y;
121  RDom r(input);
122  sum_x(x, y) = input(x, y);
123  sum_x(r.x, r.y) = sum_x(r.x, r.y) + sum_x(r.x-1, r.y);
124  sum_y(x, y) = sum_x(x, y);
125  sum_y(r.x, r.y) = sum_y(r.x, r.y) + sum_y(r.x, r.y-1);
126  \endcode
127  *
128  * You can also mix pure dimensions with reduction variables. In the
129  * previous example, note that there's no need for the y coordinate in
130  * sum_x to be traversed serially. The sum within each row is entirely
131  * independent. The rows could be computed in parallel, or in a
132  * different order, without changing the meaning. Therefore, we can
133  * instead write this definition as follows:
134  *
135  \code
136  ImageParam input(Float(32), 2);
137  Func sum_x, sum_y;
138  Var x, y;
139  RDom r(input);
140  sum_x(x, y) = input(x, y);
141  sum_x(r.x, y) = sum_x(r.x, y) + sum_x(r.x-1, y);
142  sum_y(x, y) = sum_x(x, y);
143  sum_y(x, r.y) = sum_y(x, r.y) + sum_y(x, r.y-1);
144  \endcode
145  *
146  * This lets us schedule it more flexibly. You can now parallelize the
147  * update step of sum_x over y by calling:
148  \code
149  sum_x.update().parallel(y).
150  \endcode
151  *
152  * Note that calling sum_x.parallel(y) only parallelizes the
153  * initialization step, and not the update step! Scheduling the update
154  * step of a reduction must be done using the handle returned by
155  * \ref Func::update(). This code parallelizes both the initialization
156  * step and the update step:
157  *
158  \code
159  sum_x.parallel(y);
160  sum_x.update().parallel(y);
161  \endcode
162  *
163  * When you mix reduction variables and pure dimensions, the reduction
164  * domain is traversed outermost. That is, for each point in the
165  * reduction domain, the inferred pure domain is traversed in its
166  * entirety. For the above example, this means that sum_x walks down
167  * the columns, and sum_y walks along the rows. This may not be
168  * cache-coherent. You may try reordering these dimensions using the
169  * schedule, but Halide will return an error if it decides that this
170  * risks changing the meaning of your function. The solution lies in
171  * clever scheduling. If we say:
172  *
173  \code
174  sum_x.compute_at(sum_y, y);
175  \endcode
176  *
177  * Then the sum in x is computed only as necessary for each scanline
178  * of the sum in y. This not only results in sum_x walking along the
179  * rows, it also improves the locality of the entire pipeline.
180  */
